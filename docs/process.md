# Process Note

## Development Approach

This project was developed using **Spec-Driven Development (SDD)** with AI assistance (Zed coding agent).

---

## Spec-Driven Development

Every feature began as a specification document before any code was written:

1. **Spec creation** — a spec was written following the project template (`ai-docs/specs/spec-template.md`)
2. **Implementation** — code was written to satisfy the spec's acceptance criteria
3. **Validation** — linters, type checker, tests, and E2E tests were run
4. **Review** — code review was performed against the spec

All specs are stored in `ai-docs/specs/`.

**Phases implemented (in order):**

1. Project setup (Docker, Fastify, Drizzle, TypeScript)
2. Data model (zoning + parcels tables, PostGIS, GiST indexes)
3. ArcGIS client (HTTP pagination, typed collections)
4. Ingestion pipeline (normalization, WKT conversion, UPSERT)
5. REST API (residential parcels, statistics)
6. Average lot size query (stretch goal)
7. Testing (Vitest unit + Playwright E2E)
8. Linting (ESLint, Prettier, Biome)
9. Documentation (design docs, process note, README)

## Validation

AI-generated code was validated through multiple layers:

### Automated Checks

| Check                  | Tool           | Result                          |
| ---------------------- | -------------- | ------------------------------- |
| TypeScript compilation | `tsc --noEmit` | Zero errors                     |
| Linting                | ESLint         | Zero errors, zero warnings      |
| Formatting             | Prettier       | 37 files formatted              |
| Static analysis        | Biome          | Zero errors                     |
| Unit tests             | Vitest         | 66 tests passing (12 files)     |
| E2E tests              | Playwright     | 11 tests passing (2 files)      |
| Code coverage          | Vitest + v8    | 100% statements on tested files |

### Manual Verification

- **Field name discovery**: The ArcGIS API was queried directly to discover actual field names (`Zoning_Category`, `Zoning_Description`, `Land_Use`), which differed from initial assumptions
- **SRID discovery**: Queried ArcGIS service metadata (`?f=json`) to find the `spatialReference` — zoning uses 2277, parcels uses 2278
- **Geometry format inspection**: Inspected raw API response to confirm `rings` format and coordinate order (x=lng, y=lat)
- **End-to-end test**: Ran actual ingestion against live APIs — verified 126 zoning records and 103,168 parcels imported
- **Idempotency test**: Re-ran ingestion — record counts remained identical, confirming UPSERT works
- **API response validation**: Tested all three endpoints with real data to verify SQL correctness

### AI Output Verification Process

For every code change generated by AI, the following manual review was applied:

1. **Correctness**: Does the output match the spec?
2. **API accuracy**: Are field names and SRIDs correct for the actual ArcGIS services?
3. **Edge cases**: Are null geometries, missing attributes, and empty datasets handled?
4. **Type safety**: Does the TypeScript compiler pass?
5. **Test adequacy**: Do the tests cover happy path, error path, and edge cases?

### Key Human Decisions

These were **not** delegated to AI:

| Decision                       | Rationale                                            |
| ------------------------------ | ---------------------------------------------------- |
| SRID mapping (2277 vs 2278)    | Required querying live API metadata                  |
| Field name mapping             | Required inspecting actual API responses             |
| WKT vs GeoJSON format          | Chose WKT for simplicity with `ST_GeomFromText`      |
| Error handling strategy        | Chose skip + continue (not fail-fast) for ingestion  |
| "Group by area" interpretation | Chose zoning category as the most defensible default |
